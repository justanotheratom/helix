Helix — Compile/Eval Job Runner Service

Local, docker-compose-based replacement for launch_compile_eval.py. One port (127.0.0.1:7000). One CLI. One UI. Postgres-durable queue. OpenAPI-defined contract. Langfuse fronted under /langfuse with silent auto-SSO.

Why Helix?

DSPy compile + eval today is a hand-rolled launcher writing logs to /tmp and results into the git tree. Helix replaces it with one local service: durable jobs, blob-stored artifacts, a UI, Langfuse traces deep-linked per run.

Goals

  • One URL, one CLI, one source of truth.
  • First-class concepts: program, version, dataset, split, compile→eval lineage.
  • Cancelable, restartable, durable jobs.
  • Trace correlation: every job carries a Langfuse environment tag.
  • compile.py / evaluate.py stay unchanged — they're baked into the worker image.

Non-goals (v1)

  • Multi-user auth / remote hosting.
  • Run comparison UI (defer; deep-link to EVAL_SUMMARY.md).
  • Artifact retention policies — keep everything.
  • Rewriting compile.py / evaluate.py.

User stories

  1. helix bootstrap once → helix up.
  2. helix submit compile <yaml>… returns {job_id, run_label, ui_url, langfuse_url} per job.
  3. helix list --program X filters by program/version/dataset/split/status.
  4. Click "Open in Langfuse" from job detail — only that job's traces are visible.
  5. helix cancel <id> — SIGTERM the process group, SIGKILL after 5s, keep partial artifacts.
  6. helix submit eval <yaml> --compile-job <uuid> — Helix injects the parent's program.pkl; lineage recorded.

Auto-SSO: how no-login Helix talks to a login-required Langfuse

Langfuse-web requires a NextAuth session. Helix mints it server-side so the user never sees a Langfuse login screen.
sequenceDiagram autonumber participant U as User browser (127.0.0.1:7000) participant C as caddy participant API as helix-api participant LFW as langfuse-web U->>C: GET /api/langfuse/sso?return_to=/langfuse/project/.../traces?environment=run_xyz C->>API: forward API->>LFW: POST /api/auth/callback/credentials
(seeded LANGFUSE_INIT_USER creds, compose-internal) LFW-->>API: 200 + Set-Cookie next-auth.session-token API-->>C: 302 Location: /langfuse/.../traces?...
Set-Cookie next-auth.session-token C-->>U: 302 + Set-Cookie U->>C: GET /langfuse/.../traces (carrying cookie) C->>LFW: forward (authenticated) LFW-->>U: traces page (already logged in)
  • Credentials never leave the stack. They live in the compose .env only.
  • Silent re-mint on every click — the SSO endpoint always POSTs fresh, so cookie expiry / cleared browser state never surface a login screen.
  • Same origin is what makes it work. Because /api/* and /langfuse/* are both served from 127.0.0.1:7000, the Set-Cookie attached to the 302 is in scope when the browser follows it.

Job lifecycle

Every job moves through these states. Transitions are conditional on a fencing token (worker_id + attempt) so a stale process can't corrupt a job that has been requeued.
stateDiagram-v2 [*] --> queued: helix submit queued --> running: worker UPDATE … SKIP LOCKED RETURNING running --> running: heartbeat extends lease_expires_at every 5s running --> succeeded: subprocess exit 0, artifacts uploaded running --> failed: subprocess exit != 0 running --> cancelled: SIGTERM pg → SIGKILL pg after 5s running --> queued: lease expired / worker dead → recovery requeue (attempt++) succeeded --> [*] failed --> [*] cancelled --> [*]

Cancel sequence

pgid = os.getpgid(proc.pid)         # == proc.pid because start_new_session=True
os.killpg(pgid, signal.SIGTERM)
try:
    proc.wait(timeout=5)
except subprocess.TimeoutExpired:
    os.killpg(pgid, signal.SIGKILL)
    proc.wait()
Process-group kill catches DSPy/litellm child processes that would orphan under a raw os.kill(pid, …).

Recovery (lease expired)

UPDATE jobs
SET status='queued', worker_id=NULL, lease_expires_at=NULL
WHERE status='running'
  AND (lease_expires_at < now()
       OR worker_id NOT IN (
         SELECT worker_id FROM worker_heartbeats
         WHERE last_seen > now() - interval '30 seconds'
       ));

The overlay bundle in one paragraph

The CLI ships only what's changed under backend/ai/programs/ (the active iteration surface), excluding results/**. Any dirt under baked-only roots (backend/ai/api/, backend/ai/runtime/, .claude/skills/ai-utils/, …) is a hard error pointing the user at helix up --rebuild. Deletions and renames are also hard errors in v1 (tombstones are deferred). If your tree is clean against baked_sha, the bundle is just the config YAML.

The trick that makes Helix work without forking compile.py / evaluate.py: bake the whole repo into the worker image, mount it read-only via overlayfs per job, write only into the tmpfs upperdir.

Why overlayfs, not cp -al?

cp -al hardlinks; a Python open(path,'w') truncates the inode in place and corrupts the shared snapshot. cp -R is the safe fallback if the container lacks CAP_SYS_ADMIN. Never cp -al.

Why one shared SHA across api+worker+ui?

All Helix images COPY from the same git archive <sha> tarball produced by the build script; the script writes HELIX_BAKED_REPO_SHA into each image's env. The CLI cannot diff against the wrong SHA.

Baked-only roots (changes require helix up --rebuild)

  • .claude/skills/ai-utils/ — compile.py, evaluate.py, tracing.py, helix itself
  • backend/ai/api/, backend/ai/runtime/, backend/ai/deploy/, backend/ai/scripts/
  • backend/ai/pyproject.toml, backend/ai/uv.lock, backend/ai/Dockerfile
  • Rule of thumb: anything under backend/ai/ that is NOT backend/ai/programs/.

Overlayable root (no rebuild needed)

backend/ai/programs/ in full (cross-version deps like v06/metrics.py loading v05 metrics work), except backend/ai/programs/*/*/results/** which is excluded from every overlay query.

Why partial unique indexes?

CREATE UNIQUE INDEX uq_jobs_compile_emitted
  ON jobs (program_version_id, emitted_run_number)
  WHERE type = 'compile' AND emitted_run_number IS NOT NULL;
CREATE UNIQUE INDEX uq_jobs_eval_emitted
  ON jobs (parent_job_id, emitted_run_number)
  WHERE type = 'eval' AND emitted_run_number IS NOT NULL;
-- same shape for export_run_number

Belt-and-braces against any future allocator bug. The allocator is the primary correctness mechanism; the constraints are a backstop.

Legacy run numbering (the trickiest part)

Two compile jobs can run concurrently in isolated overlays. compile.py picks max(results)+1. Without coordination they'd both claim 0033. Solution: DB-allocated number + seed placeholder dirs so compile.py deterministically picks the allocated number.
sequenceDiagram autonumber participant W as helix-worker participant DB as Postgres participant FS as overlay tmpfs participant PY as compile.py Note over W,DB: First-ever job for (program_version): allocator row absent W->>DB: INSERT INTO legacy_compile_run_numbers
VALUES ($pvid, max(observed)+2)
ON CONFLICT DO UPDATE SET next_number = next_number + 1
RETURNING next_number - 1 DB-->>W: N (the allocated number, = max+1) W->>FS: mkdir results/0001/ … results/000(N-1)/ W->>PY: spawn compile.py --config … PY->>FS: max(results)+1 == N → mkdir results/N/ PY-->>W: exit 0 W->>W: assert actual emitted dir == N (defensive guard) W->>DB: UPDATE jobs SET emitted_run_number=N (fenced)

Guarantees & acknowledged risks

guaranteed Durability

Postgres is the only durable queue. Redis carries only transient log fan-out + cancel signals. A redis outage cannot lose a job; a missed cancel just runs to completion.

guaranteed Stale-writer safety

Every post-claim Postgres write is fenced by WHERE worker_id=$me AND attempt=$attempt. A worker whose lease expired silently no-ops; its partial blob writes become unreferenced garbage.

guaranteed Reproducibility

Every job stores baked_sha. helix up refuses to build a worker image with dirty baked roots; the image is built from git archive <sha>, not the worktree.

guaranteed Artifact fidelity

Worker explicitly produces program.hash + root config copy (compile) and EVAL_SUMMARY.md (eval). Every uploaded file carries relative_path+sha256; helix export reconstructs the full tree (incl. gepa_logs/**) and skips unchanged files.

deferred to v2 Tombstones

v1 hard-errors on deletions/renames under overlayable roots. Submit-side hint: commit first, then helix up --rebuild.

accepted Pinned langfuse-web image

v1 requires a langfuse-web image built with NEXT_PUBLIC_BASE_PATH=/langfuse so subpath hosting works (static assets, NextAuth redirects). Either an upstream tag known to ship with it, or a ~10-line local layer that rebuilds with the flag.

accepted Rebuild on baked-root edits

Iterating on compile.py / evaluate.py / tracing.py requires helix up --rebuild. Iteration on backend/ai/programs/… does not. The CLI hard-errors on dirty baked roots so silent inconsistencies can't happen.

v1 scope cutout Auth, multi-user, retention

Everything bound to 127.0.0.1. No auth. Artifacts kept forever (manual docker volume rm to prune).
